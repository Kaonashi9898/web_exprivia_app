COMUNICAZIONE
Come comunicano i servizi tra loro?
Ci sono due modalità principali:
Sincrona (HTTP) — un servizio chiama direttamente un altro e aspetta la risposta.

Analogia: telefoni a qualcuno e aspetti che risponda.

prenotazioni-service → chiama → location-service
"questa postazione esiste ed è disponibile?"
Asincrona (Message Broker, es. Kafka o RabbitMQ) — un servizio pubblica un evento e gli altri lo ascoltano, senza aspettare.

Analogia: mandi un messaggio in una chat di gruppo — chi è interessato lo legge quando vuole.

prenotazioni-service → pubblica → "PRENOTAZIONE_CONFERMATA"
notification-service → ascolta → manda email/notifica
Per la tua app userei entrambe:

HTTP per le chiamate che richiedono una risposta immediata (es. verificare disponibilità postazione)
Asincrono per le notifiche

3. Come gestisco l'autenticazione tra servizi?
Qui entra in gioco il JWT. Il flusso è:
1. Utente fa login su utenti-service
2. utenti-service genera un JWT token
3. Utente manda quel token ad ogni richiesta
4. Ogni servizio verifica il token autonomamente
   senza dover chiamare utenti-service ogni volta

Analogia: è come un badge aziendale con foto e ruolo stampati sopra. Ogni ufficio lo legge da solo senza chiamare la segreteria.

Quindi ogni servizio avrà una configurazione Spring Security che valida il JWT in ingresso.

Detto tutto questo — prima di buttarci nel codice dei microservizi, c'è una domanda importante:
Serve un API Gateway?
Senza API Gateway il frontend dovrebbe conoscere l'indirizzo di ogni singolo servizio. Con un API Gateway hai un unico punto di ingresso che smista le richieste.
Frontend → API Gateway → utenti-service
                       → location-service
                       → prenotazioni-service
                       → notification-service
Per Spring Boot la soluzione naturale è Spring Cloud Gateway


per la sicurezza: 
/api/auth/register e /api/auth/login → chiunque può chiamarli (ovviamente, altrimenti non potresti mai loggarti)
/api/utenti e qualsiasi altro endpoint → devi passare il token JWT nell'header, altrimenti ricevi 403 Forbidden.


Operazione	  Metodo	           URL
Lista tutti	   GET	      /api/utenti
Cerca per ID	GET	      /api/utenti/1
Modifica	      PUT	      /api/utenti/1 + body {"fullName":"...", "email":"..."}
Elimina	      DELETE	   /api/utenti/1
Per la PUT il body accetta solo fullName ed email (non si può cambiare ruolo o password da qui, che è corretto).


AUTENTICAZIONE 
Con utente USER → GET /api/utenti deve dare 403 Forbidden

Con utente ADMIN → deve funzionare normalmente

Per creare un utente ADMIN, nella POST /api/auth/register aggiungi il campo ruolo:


{
  "fullName": "Admin Test",
  "email": "admin@example.com",
  "password": "Password123!",
  "ruolo": "ADMIN"
}
Concetto chiave: Spring Security richiede il prefisso ROLE_ internamente. 
Noi lo aggiungiamo nel filtro ("ROLE_" + ruolo), e nella config usiamo 
hasRole("ADMIN") che cerca automaticamente ROLE_ADMIN. 
Le due parti si incastrano.

PER I GRUPPI 
tabella di collegamento gruppi_utente che mette in relazione utenti e gruppi
(relazione molti-a-molti: un utente può stare in più gruppi, un gruppo può avere più utenti).

Operazione	               Metodo	URL
Crea un gruppo	            POST	   /api/gruppi
Lista tutti i gruppi	      GET	   /api/gruppi
Aggiungi utente a gruppo	POST	   /api/gruppi/{idGruppo}/utenti/{idUtente}
Rimuovi utente da gruppo	DELETE	/api/gruppi/{idGruppo}/utenti/{idUtente}
Utenti di un gruppo	      GET	   /api/gruppi/{idGruppo}/utenti

AD ESEMPIO:
# 1. Crea un gruppo
POST /api/gruppi?nome=Sviluppo

# 2. Lista gruppi
GET /api/gruppi

# 3. Aggiungi Mario (id=1) al gruppo (id=1)
POST /api/gruppi/1/utenti/1

# 4. Vedi gli utenti del gruppo
GET /api/gruppi/1/utenti

# 5. Rimuovi Mario dal gruppo
DELETE /api/gruppi/1/utenti/1

Tutti gli endpoint richiedono il token di un utente ADMIN nell'header.
